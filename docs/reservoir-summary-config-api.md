# Reservoir-summary config API (для фронта)

Конфиг определяет, **какие** организации попадают в отчёт
`GET /reservoir-summary?date=...`, **в каком порядке** они идут, и **чьи
данные** суммируются в строку **ИТОГО**.

Источник правды для всех каналов вывода:

- JSON-эндпоинт `GET /reservoir-summary?date=...`.
- Excel-экспорт `GET /reservoir-summary/export?date=...`.
- Excel-экспорт `GET /filter/export?date=...` (использует тот же шаблон с
  блоками фильтрации ниже).
- Excel-экспорт `GET /manual-comparison/export?date=...`.

## Эндпоинты

| Метод | Путь | Роли | Описание |
|---|---|---|---|
| `GET` | `/reservoir-summary/config` | `sc`, `rais`, `cascade` | Получить список всех конфигов |
| `POST` | `/reservoir-summary/config` | `sc`, `rais` | Создать или обновить запись (upsert по `organization_id`) |
| `DELETE` | `/reservoir-summary/config?organization_id={id}` | `sc`, `rais` | Удалить запись |

`cascade`-роль видит только записи, привязанные к её собственным
организациям; `sc`/`rais` видят всё.

## DTO

### Request — POST body

```ts
interface UpsertReservoirSummaryConfigRequest {
  organization_id: number;    // required, > 0
  sort_order: number;         // required, >= 0; позиция в отчёте
  include_in_total: boolean;  // суммировать ли значения в строку ИТОГО
  modsnow_enabled: boolean;   // default true для новых записей; см. ниже
}
```

> Фронт должен **всегда** передавать `modsnow_enabled` — бэкенд читает
> JSON через `render.DecodeJSON` без явного дефолта, и если поле
> отсутствует, в базу запишется `false` (zero-value Go-bool). Для новых
> записей нормальное значение — `true`. См. раздел
> «Поведение `modsnow_enabled`» — миграция 000085 проставила `true`
> всем существующим организациям отчёта, кроме одной (см. раздел).

### Response — GET (массив)

```ts
interface ReservoirSummaryConfig {
  id: number;
  organization_id: number;
  organization_name: string;   // JOIN с organizations.name, для UI
  sort_order: number;
  include_in_total: boolean;
  modsnow_enabled: boolean;
}
```

### Коды ответов

| Эндпоинт | Успех | Ошибки |
|---|---|---|
| `POST` | `200 OK` + `{"status":"OK"}` | `400` invalid JSON / validation; `500` БД |
| `GET` | `200 OK` + массив `ReservoirSummaryConfig[]` (может быть пустым) | `500` БД |
| `DELETE` | `204 No Content` | `400` missing/invalid `organization_id`; `404` config not found; `500` БД |

## Семантика полей

- **`sort_order`** — позиция объекта в массиве ответа
  `GET /reservoir-summary?date=...` и в строках Excel-отчёта. Бэк не
  валидирует уникальность — если два конфига имеют одинаковый
  `sort_order`, порядок между ними недетерминирован. **Фронт обязан**
  поддерживать уникальность.
- **`include_in_total`** — `true` → значения объекта (volume, income,
  release, modsnow, incoming_volume) попадают в SQL-строку `ИТОГО`.
  `false` → объект в отчёте показан, но в `ИТОГО` не суммируется.
  Используется для объектов, визуально расположенных НИЖЕ строки ЖАМИ
  в макете (Пском).
- **`modsnow_enabled`** — гейт по организации для значения modsnow в
  обоих каналах (JSON и Excel). См. раздел «Поведение `modsnow_enabled`»
  ниже. Default в БД — `true`.
- **Whitelist** — если организация **отсутствует** в
  `reservoir_summary_config`, она **не появится** в `/reservoir-summary`,
  даже если для неё есть данные в `reservoir_data`. Это фича.

## Порядок ИТОГО-строки

`ИТОГО` встаёт в массиве **после** последнего объекта с
`include_in_total = true` и **перед** первым объектом с
`include_in_total = false`. Логика:

```
sort_position = sort_order * 10                         (для каждого org'а)
sort_position = max(sort_order WHERE include_in_total) * 10 + 5   (для ИТОГО)
```

Пример для текущего seed'а (8 объектов):

```
sort_order  include_in_total   объект              sort_position
1           true               Андижон             10
2           true               Охангарон           20
3           true               Сардоба             30
4           true               Хисорак             40
5           true               Топаланг            50
6           true               Чорвок              60
7           true               Қуйи Чоткол         70
—           —                  ИТОГО               75   ← между 7 и 8
8           false              Пском               80
```

## Эффект на существующий `/reservoir-summary?date=...`

После деплоя возможны **breaking changes** для фронта:

- **Порядок:** больше не строковый match по имени в Go-коде, а
  `ORDER BY sort_position` из БД. Объекты, чьи имена раньше не
  матчились (Пском, Қуйи Чоткол), теперь имеют детерминированную
  позицию.
- **Состав:** объект `Қуйи Чоткол` теперь СТАБИЛЬНО входит в ИТОГО
  (`include_in_total = true`); объект `Пском` СТАБИЛЬНО исключён
  (`include_in_total = false`). До этого PR'а оба были вне ИТОГО для
  `volume`/`income`/`release`/`modsnow` (из-за рассинхрона имён) и
  оба были ВНУТРИ ИТОГО для `incoming_volume` (там не было фильтра).
- **`incoming_volume` в ИТОГО** теперь выровнен с остальными полями —
  тоже фильтруется по `include_in_total`. Цифра может сместиться.

## Поведение `modsnow_enabled`

Флаг `modsnow_enabled` управляет тем, показывается ли значение modsnow
для данной организации **в обоих каналах** — JSON и Excel.

| `modsnow_enabled` | JSON-ответ `/reservoir-summary` | Excel-ячейка modsnow | Фронт |
|---|---|---|---|
| `true`  | `Modsnow.current` / `Modsnow.year_ago` приходят из таблицы `modsnow` как есть | в `N6..N22` / `O6..O22` пишется текущее и прошлогоднее значение | поле редактируемо как обычно |
| `false` | `Modsnow.current = 0` и `Modsnow.year_ago = 0` (значения маскируются на бэке независимо от того, что в БД) | ячейка явно очищается — не остаётся ни данных, ни шаблонного значения | поле должно быть read-only / скрыто |

Миграция 000085 проставила `modsnow_enabled = true` всем существующим
организациям отчёта, кроме одной, для которой modsnow раньше
скрывалась в Excel-генераторе через скип по позиционному индексу
(теперь это решение перенесено в БД).

Семантика:

- Маскировка применяется только к колонкам modsnow (`current`,
  `year_ago`). Поле `two_years_ago` в отчёте не отображается и не
  маскируется.
- Если в `reservoir_summary_config` нет записи для организации
  (через JSON-эндпоинт это невозможно, но теоретически — прямой
  INSERT в `reservoir_data`), JSON-маскировка не срабатывает,
  а Excel-генератор в строгом режиме оставляет ячейку пустой.
- Если БД-запрос на список конфигов упал (`500`), бэк продолжает
  отвечать, но маскирует modsnow для **всех** организаций —
  безопаснее, чем случайно показать данные у тех, кому флаг
  стоял в `false`.

## ⚠️ Жёсткая связь с Excel-шаблоном

Excel-генератор работает по **позиционным слотам** шаблона
`res-summary.xlsx` (rows 6, 8, 10, 12, 14, 16, 18, 22). Слот определяется
индексом в отсортированном массиве, который репо возвращает по
`sort_order`. Это значит:

| Sort_order | Слот в Excel | Объект из текущего seed'а | Особенность |
|---|---|---|---|
| 1 | row 6/7   | Андижон     | — |
| 2 | row 8/9   | Охангарон   | — |
| 3 | row 10/11 | (см. seed)  | `modsnow_enabled = false` — ячейки modsnow N10/O10 остаются пустыми |
| 4 | row 12/13 | Ҳисорак     | — |
| 5 | row 14/15 | Тўполанг    | — |
| 6 | row 16/17 | Чорвоқ      | — |
| 7 | row 18/19 | Қуйи Чоткол | Включается в ЖАМИ (формулы 20/21) |
| 8 | row 22/23 | Пском       | `include_in_total = false` — НИЖЕ ЖАМИ в макете |

Из этого правила:

- **Не вставляйте новый объект между sort_order=1..7**, не передвинув
  существующие — иначе старшие сместятся в неправильные слоты шаблона.
- **9-я запись игнорируется** — Excel шаблон имеет ровно 8 слотов.
- Поведение modsnow больше **не** зависит от позиции в шаблоне —
  раньше пропуск modsnow на 3-м слоте был захардкожен в Go-генераторе,
  теперь это решение перенесено в БД через `modsnow_enabled`. Можно
  менять `sort_order` у любого объекта без правки генератора.

## UI-рекомендации

- **Capacity = 8.** Шаблон `res-summary.xlsx` имеет ровно 8 слотов под
  объекты. Если в конфиге 9-я запись, Excel-отчёт молча уронит её. Фронт
  должен блокировать создание 9-й или предупреждать админа.
- **Drag-and-drop порядок.** `sort_order` удобнее редактировать
  перетаскиванием строк (фронт пересохраняет POST'ы для затронутых
  строк).
- **Чек-бокс "Суммировать в ИТОГО"** рядом со строкой = `include_in_total`.
- **Удаление организации** не освобождает её `sort_order` — следующие
  записи **не** сдвигаются вверх. Если нужно — фронт делает несколько
  POST'ов для пересчёта.
