# POST /api/v3/reservoir-summary — частичные апдейты

Все числовые поля в item-е поддерживают **трёх-значный** контракт:

| JSON payload            | На существующую строку                      | На новую строку           |
|-------------------------|----------------------------------------------|---------------------------|
| поле **отсутствует**    | колонка **не меняется**                      | пишется `0`               |
| поле равно **`null`**   | пишется `0` (или `NULL` для totals-колонок)  | пишется `0` (или `NULL` для totals-колонок) |
| поле — **число**        | пишется это число                            | пишется это число         |

Контракт распространяется на:

- `income`
- `level`
- `release`
- `volume`
- `total_income_volume` *(колонка nullable → `null` пишет `NULL`, не `0`)*
- `total_income_volume_prev_year` *(то же самое)*

**Обязательные** поля (должны присутствовать всегда): `organization_id`, `date` (формат `YYYY-MM-DD`).

**Не попадают в контракт:**

- `modsnow_current`, `modsnow_year_ago` — живут в отдельной таблице `modsnow`. `null` или отсутствие = no-op; только число пишет значение.

## Примеры

### Частичный апдейт — меняется только `income`

```json
[
  {"organization_id": 1, "date": "2026-04-12", "income": 99}
]
```

Результат: `income_m3_s` становится `99`, остальные колонки не тронуты.

### Сбросить `release` в ноль

```json
[
  {"organization_id": 1, "date": "2026-04-12", "release": null}
]
```

Результат: `release_m3_s` становится `0`, остальные колонки не тронуты.

### Явный `0` сохраняется как `0`

```json
[
  {"organization_id": 1, "date": "2026-04-12", "level": 0}
]
```

Результат: `level_m` становится `0`. Это отличается от "поле отсутствует": если `level` не отправить вообще, старое значение в БД сохранится.

### Гайдлайн для фронта

Слать **только те поля, которые пользователь реально отредактировал**. Не слать
дефолты или «залипшие» значения из формы — они перезапишут то, что сейчас лежит в БД.
