# POST /api/v3/ges-report/daily-data — partial updates + bulk

Эндпоинт теперь принимает **массив** item-ов (даже если вы обновляете
только одну станцию — нужно завернуть объект в `[...]`). В одном запросе
можно записать данные сразу по нескольким станциям — они будут
применены атомарно в одной транзакции.

Числовые поля поддерживают **три-значный контракт**: «поле отсутствует»,
«`null`», «число». Это позволяет точечно обновлять только те поля,
которые пользователь реально отредактировал, не затирая остальные. Раньше
любой POST перезаписывал все 8 числовых полей целиком — теперь можно
слать только изменившиеся.

## Контракт для 6 nullable полей

Поля: `water_level_m`, `water_volume_mln_m3`, `water_head_m`,
`reservoir_income_m3s`, `total_outflow_m3s`, `ges_flow_m3s`.

Колонки в БД — nullable, поэтому поддерживают полный три-значный
контракт:

| JSON payload | На существующую строку | На новую строку |
| --- | --- | --- |
| поле **отсутствует** | колонка **не меняется** | пишется `NULL` |
| поле = **`null`** | пишется `NULL` | пишется `NULL` |
| поле = **число** | пишется это число | пишется это число |

Семантика: `NULL` означает «нет данных», `0` — «значение реально равно
нулю». Эти два состояния **различаются**.

## Контракт для 2 NOT NULL полей

Поля: `daily_production_mln_kwh`, `working_aggregates`. Колонки в БД —
`NOT NULL DEFAULT 0`, поэтому `null` физически не может быть записан:

| JSON payload | На существующую строку | На новую строку |
| --- | --- | --- |
| поле **отсутствует** | колонка **не меняется** | пишется `0` (DEFAULT) |
| поле = **`null`** | пишется `0` (через COALESCE) | пишется `0` |
| поле = **число** | пишется это число | пишется это число |

**NB:** для этих двух полей `null` неотличимо от `0` — это вынужденная
асимметрия из-за схемы БД. Если потребуется отличать «нет данных» от
«реально 0» — это будущая отдельная задача (потребуется `ALTER TABLE`
снять `NOT NULL`).

## Обязательные поля

В каждом item-е массива **обязательно** должны присутствовать:

- `organization_id` (`int64`) — id станции
- `date` (`string`, формат `YYYY-MM-DD`) — операционный день

Без них запрос отклоняется с `400 Bad Request`.

## Bulk: пример массива

```json
[
  {
    "organization_id": 100,
    "date": "2026-04-13",
    "daily_production_mln_kwh": 8.4,
    "working_aggregates": 3,
    "water_level_m": 890.5
  },
  {
    "organization_id": 101,
    "date": "2026-04-13",
    "daily_production_mln_kwh": 5.2,
    "working_aggregates": 2,
    "water_level_m": 712.0
  }
]
```

Транзакция **атомарная**: либо все станции записаны, либо ни одна. Если
у одного item-а ошибка валидации — весь запрос отклоняется с указанием
`item_index` (0-индексированный) в теле ответа.

## Примеры частичных апдейтов

### Поправить только `water_level_m` у одной станции

```json
[
  {"organization_id": 100, "date": "2026-04-13", "water_level_m": 891.0}
]
```

Результат: в `ges_daily_data` обновится только `water_level_m`.
Остальные 7 полей не тронуты.

### Сбросить `reservoir_income_m3s` в `NULL` (нет данных)

```json
[
  {"organization_id": 100, "date": "2026-04-13", "reservoir_income_m3s": null}
]
```

Результат: `reservoir_income_m3s = NULL`. Остальные поля не тронуты.

### Явный `0` для NOT NULL поля

```json
[
  {"organization_id": 100, "date": "2026-04-13", "ges_flow_m3s": 0}
]
```

Результат: `ges_flow_m3s = 0`. Это **отличается** от «не прислано» —
если бы поле не было прислано, сохранилось бы предыдущее значение.

## Гайдлайн для фронта

- Слать **только те поля, которые пользователь реально отредактировал**.
  Не слать дефолты или «залипшие» значения из формы — они либо
  перезатрут существующие данные, либо запишут `NULL`/`0` где не надо.
- Перед редактированием сделайте
  `GET /api/v3/ges-report/daily-data?organization_id=...&date=...` для
  предзаполнения формы. После редактирования сравните с исходным
  снапшотом и отправьте только изменённые поля.
- Для записи нескольких станций одним запросом — соберите массив локально
  и отправьте одним POST. Не нужно делать отдельный запрос на каждую
  станцию.

## Ошибки

- `400 Bad Request` — пустой массив, отсутствуют `organization_id`/`date`,
  неправильный формат даты. Тело ответа содержит `item_index`
  (0-индексированный) для ошибок валидации внутри массива.
- `401 Unauthorized` — нет токена.
- `403 Forbidden` — токен есть, но роль не `sc`/`rais`, либо нет доступа
  к одной из `organization_id` в массиве (батч проверяется целиком — если
  хоть одна станция недоступна, отклоняется весь запрос).
- `500 Internal Server Error` — ошибка БД.

## Что НЕ принимается

Поля `temperature` и `weather_condition` **не входят** в DTO этого
эндпоинта. Погода пишется автоматически фоновым тикером в таблицу
`cascade_daily_data` на уровне каскада. Подробнее —
[docs/ges-cascade-weather.md](ges-cascade-weather.md).
