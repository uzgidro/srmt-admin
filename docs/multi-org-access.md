# Мульти-организационный доступ — изменения для фронтенда

Пользователь теперь может быть привязан **к нескольким организациям** (раньше —
ровно к одной). Это позволяет одному профилю дежурного видеть и свой каскад ГЭС,
и своё водохранилище — без второго аккаунта.

## 1. Новый эндпоинт: привязка организаций к пользователю

`PUT /users/{userID}/organizations` — заменяет весь список организаций пользователя.

Доступ: только роль `admin` (та же группа, что `POST /users/{userID}/roles`).

**Тело запроса:**

```json
{ "organization_ids": [5, 10] }
```

- `organization_ids` — **обязательное** поле. `null` или отсутствие поля → `400`.
- Пустой массив `[]` — допустим, означает «снять все привязки».
- Дубли в массиве отбрасываются на стороне сервера.

Эндпоинт — **полная замена** списка (идемпотентный PUT). При сохранении формы
редактирования пользователя отправляйте `PUT .../organizations` с текущим
содержимым поля **всегда** — включая пустой `[]`. Сравнивать с прежним списком
и слать только при изменении не нужно: серверная операция дешёвая, а «всегда
слать» исключает рассинхрон и ветвление на стороне фронта.

**Ответы:**

| Код | Когда |
|-----|-------|
| `204 No Content` | Успех — список заменён. |
| `400 Bad Request` | Невалидный `userID` в пути; отсутствует `organization_ids`; неизвестный пользователь или организация (`{"error":"user or organization not found"}`). |
| `401 Unauthorized` | Нет/невалидный токен. |
| `403 Forbidden` | Роль не `admin`. |
| `500` | Внутренняя ошибка. |

### Сценарий создания объединённого аккаунта

1. **`POST /users`** — создать пользователя. Роли передаются прямо в теле
   (`role_ids`, обязательное, минимум одна — например `cascade`, `reservoir`,
   `reservoir_flood`). Ответ: **`201 Created`**, тело `{"id": <userID>}` —
   берём `id` из ответа.
2. **`PUT /users/{id}/organizations`** — задать привязку к нескольким
   организациям (каскад + водохранилище). Мульти-орг привязка в `POST /users`
   не закладывается: там `contact.organization_id` — одна, основная.
3. `POST /users/{id}/roles` нужен только если позже меняется набор ролей
   (при создании роли уже заданы в шаге 1).

## 2. Новое поле в ответах о пользователе

`GET /users` и `GET /users/{id}` теперь возвращают поле `organization_ids`:

```json
{
  "id": 42,
  "login": "duty1",
  "organization_ids": [5, 10],
  "roles": ["cascade", "reservoir", "reservoir_flood"],
  ...
}
```

`organization_ids` — массив `int64`. Пустой `[]` у пользователя без привязок.
Поле `organization` (одиночная организация из контакта) сохранено как было.

## 3. Переименование роли

Роль `reservoir_duty` переименована в **`reservoir_flood`** (дежурный
водохранилища — паводковый журнал). При выдаче ролей через
`POST /users/{id}/roles` используйте новое имя `reservoir_flood`.

Объединённый аккаунт дежурного получает три роли:
`cascade` + `reservoir` + `reservoir_flood`.

## 4. JWT: claim `org_id` → `org_ids`

Если фронт парсит JWT — поле `org_id` (число) заменено на `org_ids` (массив
чисел). Для большинства фронтов это незаметно (токен не парсится вручную).

## 5. Эффект деплоя

Сразу после деплоя **выданные ранее access-токены теряют организацию** (старое
поле `org_id` не читается, `org_ids` будет пустым). Последствие:

- Пользователи с ролью `sc`/`rais` — без эффекта (полный доступ не зависит от организаций).
- Пользователи с ролями `cascade`/`reservoir`/`reservoir_flood` — до обновления
  токена получат `403` на эндпоинтах, ограниченных организацией.

Лечится автоматически: первый же вызов `POST /auth/refresh` выдаёт новый
access-токен с корректным `org_ids` (refresh-токен не затронут). Пользователь
не разлогинивается. Если фронт делает silent-refresh по `401`/`403` — никаких
действий не нужно; иначе стоит инициировать refresh после деплоя.
