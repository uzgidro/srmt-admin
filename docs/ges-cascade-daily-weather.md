# POST/GET /ges-report/cascade-daily-data — ручная коррекция погоды каскада

Таблица `cascade_daily_data` обычно заполняется автоматически фоновым
планировщиком в 04:00 Asia/Tashkent (`dayrotation.Service.fetchWeather`) —
он снимает погоду через OpenWeatherMap по координатам из `cascade_config`
и апсертит результат для каждого каскада. Этот эндпоинт нужен для
**ручной коррекции**, когда данные OpenWeatherMap неверны или
недоступны.

**Важный caveat:** планировщик на следующем cutoff (04:00 Asia/Tashkent)
перезапишет ручную коррекцию. Ручная правка — для «починить погоду
текущего дня пока тикер следующий раз не сработает», не для долгосрочного
хранения.

Смежные доки:

- [ges-cascade-weather.md](ges-cascade-weather.md) — автоматический flow
  и схема `cascade_daily_data`.
- [ges-daily-data-partial-update.md](ges-daily-data-partial-update.md) —
  станционный partial-update контракт (для сравнения: там есть

## POST контракт

Body всегда **массив**, даже если корректируется один каскад. Пример:

```json
[
  {
    "organization_id": 10,
    "date": "2026-04-13",
    "temperature": 22.5,
    "weather_condition": "01d"
  }
]
```

Обязательные поля в каждом item-е:

- `organization_id` (`int64`) — должна быть **каскадная** организация,
  т.е. иметь запись в `cascade_config`.
- `date` (`string`, формат `YYYY-MM-DD`) — операционный день.

## Контракт для `temperature` и `weather_condition`

Обе колонки в БД — nullable (`temperature NUMERIC`,
`weather_condition TEXT`), поэтому поддерживают полный три-значный
контракт **без асимметрии**:

| JSON payload | На существующую строку | На новую строку |
| --- | --- | --- |
| поле **отсутствует** | колонка **не меняется** | пишется `NULL` |
| поле = **`null`** | пишется `NULL` | пишется `NULL` |
| поле = **значение** | пишется это значение | пишется это значение |

Можно точечно править только `temperature`, не трогая
`weather_condition`, и наоборот.

## Примеры POST

### Частичная коррекция — только температура

```json
[
  {"organization_id": 10, "date": "2026-04-13", "temperature": 23.0}
]
```

Результат: `temperature = 23.0`, `weather_condition` в БД не тронуто.

### Массовая коррекция на 2 каскадах

```json
[
  {"organization_id": 10, "date": "2026-04-13", "temperature": 23.0, "weather_condition": "01d"},
  {"organization_id": 20, "date": "2026-04-13", "temperature": 18.5, "weather_condition": "10d"}
]
```

Транзакция **атомарная**: либо все записаны, либо ни один. Если у одного
item-а ошибка валидации — весь запрос отклоняется с указанием
`item_index` (0-индексированный) в теле ответа.

### Сброс некорректного значения в `NULL`

```json
[
  {"organization_id": 10, "date": "2026-04-13", "weather_condition": null}
]
```

Результат: `weather_condition = NULL`. `temperature` не тронута.

## GET контракт

Query params:

- `organization_id` (`int64`, обязателен) — id каскадной организации.
- `date` (`string`, формат `YYYY-MM-DD`, обязателен) — операционный день.

Пример запроса:

```http
GET /ges-report/cascade-daily-data?organization_id=10&date=2026-04-13
```

Если запись существует:

```json
{
  "temperature": 22.5,
  "weather_condition": "01d"
}
```

Если записи за эту дату нет — ответ `null` с HTTP 200 (не 404):

```json
null
```

Так удобнее фронту для предзаполнения формы — не нужно отдельно
обрабатывать 404.

## Ошибки

- **400 Bad Request** — пустой массив, отсутствует
  `organization_id`/`date`, неправильный формат даты, `organization_id`
  не является каскадной организацией (нет в `cascade_config`). Тело
  ответа содержит `item_index` (0-индексированный) для ошибок внутри
  массива.
- **401 Unauthorized** — нет токена.
- **403 Forbidden** — роль не `sc`/`rais`, либо нет доступа хотя бы к
  одной `organization_id` из массива (батч проверяется целиком — если
  хоть одна недоступна, отклоняется весь запрос).
- **500 Internal Server Error** — ошибка БД (транзакция откачена).

## Гайдлайн для фронта

- Перед редактированием сделайте `GET
  /ges-report/cascade-daily-data?organization_id=...&date=...` —
  предзаполните форму текущими значениями.
- Отправляйте в POST **только те поля, которые пользователь реально
  отредактировал**. Отсутствующие поля не будут затёрты.
- Чтобы явно сбросить значение в `NULL` — пришлите `null`, не опускайте
  поле.
- `organization_id` должна быть **каскадной** — проверяйте через
  `GET /ges-report/cascade-config`, какие org являются каскадами.
- Помните про перезапись тикером: ручная коррекция держится до
  следующего 04:00 Asia/Tashkent. Если нужно «залочить» значение
  надолго — это отдельная задача (потребуется флаг `manual_override` на
  строке).
