# Shutdowns — Cascade Owner Restriction — Frontend Implementation Guide

Документ описывает что нужно изменить на фронте после введения нового ограничения: пользователь с ролью `cascade` теперь может редактировать и удалять только собственные записи `shutdowns`. Для ролей `sc` и `rais` поведение не меняется.

## Бизнес-цель и семантика

Раньше любой пользователь с ролью `cascade`, имеющий доступ к организации (через свой каскад), мог редактировать и удалять любые `shutdowns` этой организации — даже созданные коллегами. Это приводило к конфликтам авторства и непреднамеренному переписыванию чужих записей.

**Новое правило.** Пользователь с ролью `cascade` может выполнить `PATCH` и `DELETE` только над теми `shutdowns`, у которых `created_by_user_id == claims.UserID`. Запись с `created_by_user_id IS NULL` (бывший владелец удалён, FK `ON DELETE SET NULL`) становится **read-only** для всех cascade-юзеров — её больше нельзя редактировать или удалить через cascade-роль. Доступ остаётся у `sc` и `rais`.

Создание (`POST`) не затронуто: создатель сразу становится владельцем.
Просмотр (`GET`) не затронут.
Подтверждение прочтения (`PATCH /shutdowns/{id}/viewed`) не затронуто — это операция «ack», а не редактирование.

| Роль | Было | Стало |
| --- | --- | --- |
| `sc`, `rais` | Полный доступ к любому shutdown в любой org | Без изменений — полный доступ |
| `cascade`, владелец записи | PATCH/DELETE разрешён | PATCH/DELETE разрешён |
| `cascade`, не владелец, та же org/каскад | PATCH/DELETE разрешён | **403 — нет прав** |
| `cascade`, NULL owner (бывший владелец удалён) | PATCH/DELETE разрешён | **403 — нет прав** |
| Другие роли | Без write-доступа по существующему RBAC | Без изменений |

## Какие endpoint'ы затронуты

| Endpoint | Затронут | Комментарий |
| --- | --- | --- |
| `PATCH /shutdowns/{id}` | Да | Новая 403 для cascade-юзера на чужой записи или записи с NULL owner |
| `DELETE /shutdowns/{id}` | Да | Новая 403 для cascade-юзера на чужой записи или записи с NULL owner |
| `POST /shutdowns` | Нет | Создатель сразу становится owner; ограничение неприменимо |
| `PATCH /shutdowns/{id}/viewed` | Нет | Это ack-операция, доступна по тем же правилам что и раньше |
| `GET /shutdowns` | Нет | Чтение остаётся без изменений |
| `GET /shutdowns/{id}` | Нет | Чтение остаётся без изменений |

## Семантика 403 ответов

Backend возвращает следующие тексты ошибок:

```json
{"error": "only the creator can edit this record"}
```

```json
{"error": "only the creator can delete this record"}
```

**Когда возвращается:**

- Текущая роль — `cascade` (без `sc`/`rais`).
- `created_by_user_id != claims.UserID` ИЛИ `created_by_user_id IS NULL`.

**Когда НЕ возвращается:**

- Текущая роль — `sc` или `rais` — bypass; этот 403 не вернётся никогда.
- `created_by_user_id == claims.UserID` — собственная запись.

**Отличие от существующего 403.** Уже есть отдельный 403 «Нет доступа к целевой организации», который возникает, например, когда cascade-юзер пытается переместить запись в чужую организацию через `PATCH` с другим `organization_id`. Это **разная** семантика и разный текст:

| Текст ошибки | Источник | Что значит |
| --- | --- | --- |
| `only the creator can edit this record` / `only the creator can delete this record` | Новая проверка владельца | Cascade-юзер не владеет конкретной записью |
| `access to target organization denied` (или похожий существующий) | Существующая проверка org-доступа | Cascade-юзер пытается работать с чужой организацией |

Фронт должен различать эти два случая по тексту/коду в теле ответа и показывать разные toast-сообщения.

## UI/UX рекомендации

- **Маркер чужой записи.** В таблице/списке shutdowns рядом с записями, у которых `created_by_user.id != currentUser.id`, показывать значок «не моё» (например, серый человечек). Действует только для cascade-юзеров; для sc/rais маркер не нужен.
- **Маркер orphan.** Для записей с `created_by_user == null` показывать значок «orphan» (например, перечёркнутый человечек). Это сигнал, что запись стала read-only для cascade-роли.
- **Кнопки edit/delete.** Два варианта:
  1. **Дизейблить** кнопки на чужих/orphan-записях для cascade-юзеров. Tooltip: «Только создатель записи может её изменить».
  2. **Оставить кликабельными**, но при 403 показать toast.

  Рекомендация — **второй вариант**: меньше работы (не нужно дублировать backend-логику владения на фронте), та же UX. Если на фронте уже есть permission-helper из §5 — можно безболезненно использовать первый вариант.

- **sc/rais.** Для этих ролей кнопки edit/delete всегда активны, маркеры «не моё»/orphan не показываем — у них нет ограничения по владельцу.

## TS-типы + helper

```typescript
interface User {
  id: number;
  name: string;
}

interface Shutdown {
  id: number;
  organization_id: number;
  created_by_user: User | null;  // null если бывший владелец удалён
  // ...прочие существующие поля...
}

interface CurrentUser {
  id: number;
  roles: string[];
}

// Проверка может ли текущий юзер мутировать запись.
// Используется для дизейбла кнопок и для маркеров «не моё»/orphan.
function canCurrentUserMutate(shutdown: Shutdown, currentUser: CurrentUser): boolean {
  if (currentUser.roles.includes("sc") || currentUser.roles.includes("rais")) {
    return true;
  }
  if (currentUser.roles.includes("cascade")) {
    return shutdown.created_by_user?.id === currentUser.id;
  }
  return false; // другие роли не имеют write-доступа по существующему RBAC
}
```

Если в текущем интерфейсе `Shutdown` ещё нет поля `created_by_user` — добавить его (backend уже отдаёт). Тип `User | null` обязателен.

## Обработка ошибок

| Code | Условие | Что показать |
| --- | --- | --- |
| 200 | edit OK | стандартный success |
| 200 | delete OK (тело пустое) | стандартный success |
| 403 + текст про creator | пользователь не владелец (cascade-юзер на чужой/orphan записи) | toast «Только создатель может изменить эту запись» (для PATCH) или «Только создатель может удалить эту запись» (для DELETE) |
| 403 + другой текст («Нет доступа к целевой организации» или подобное) | существующая проверка org-доступа (cascade на чужую org) | toast «Нет доступа к этой организации» |
| 404 | shutdown не существует | toast «Запись не найдена» |
| 500 | сервер | toast «Ошибка сервера» |

Различение 403-вариантов — по подстроке `creator` в `error`-тексте. Простейший псевдокод:

```typescript
if (response.status === 403) {
  const msg = response.data?.error ?? "";
  if (msg.includes("creator")) {
    showToast(method === "DELETE"
      ? "Только создатель может удалить эту запись"
      : "Только создатель может изменить эту запись");
  } else {
    showToast("Нет доступа к этой организации");
  }
  return;
}
```

## End-to-end сценарии

### Сценарий 1 — cascade-юзер 10 редактирует свою запись

1. Юзер 10 (роль `cascade`, org=42) открывает shutdown #555, у которого `created_by_user.id = 10`.
2. Меняет описание, нажимает «Сохранить».
3. Фронт: `PATCH /shutdowns/555` с обновлённым телом.
4. Backend: проверка владельца проходит (10 == 10). 200 OK.
5. UI обновляется, показывается success-toast.

### Сценарий 2 — cascade-юзер 11 пробует редактировать запись юзера 10 (тот же каскад)

1. Юзер 11 (роль `cascade`, тот же каскад/org что и юзер 10) видит в списке shutdown #555 со значком «не моё».
2. Кликает «Редактировать», меняет описание, нажимает «Сохранить».
3. Фронт: `PATCH /shutdowns/555`.
4. Backend: org-доступ есть, но `created_by_user_id (10) != claims.UserID (11)` → 403 `{"error": "only the creator can edit this record"}`.
5. UI показывает toast «Только создатель может изменить эту запись». Локальное состояние не обновляется.

### Сценарий 3 — sc-админ редактирует ту же запись

1. Sc-админ открывает shutdown #555.
2. Меняет описание, нажимает «Сохранить».
3. Фронт: `PATCH /shutdowns/555`.
4. Backend: роль `sc` — bypass проверки владельца. 200 OK.
5. UI обновляется, показывается success-toast.

### Сценарий 4 (опционально) — запись с NULL owner

1. Юзер, ранее создавший shutdown #777, удалён из системы. FK сработал: `created_by_user_id IS NULL`.
2. Любой cascade-юзер (хоть из того же каскада) пробует `PATCH /shutdowns/777` или `DELETE /shutdowns/777`.
3. Backend: 403 `{"error": "only the creator can edit this record"}` (или `...delete...` для DELETE).
4. Sc/rais — 200 OK, мутация проходит.

В UI запись помечена значком «orphan»; для cascade-юзеров кнопки edit/delete либо дизейблены, либо при клике дают тот же toast, что в сценарии 2.

## Changelog / Совместимость

- Изменение поведения только для роли `cascade`. Роли `sc` и `rais` — без изменений.
- API shape `PATCH /shutdowns/{id}` и `DELETE /shutdowns/{id}` НЕ меняется — те же payload и response, только новый возможный 403.
- Старый фронт без обновления продолжит работать. Просто иногда будет получать 403 там, где раньше получал 200. Если в коде есть generic-обработчик 403 — отображается как «Нет доступа» (не критично, но не информативно).

### Чек-лист готовности фронта

- [ ] В TS-интерфейсе `Shutdown` есть поле `created_by_user: User | null` (если ещё не было).
- [ ] Реализован helper `canCurrentUserMutate(shutdown, currentUser)` (см. §5).
- [ ] В списке/таблице shutdowns для cascade-юзеров показывается значок «не моё» на чужих записях и значок «orphan» на записях с `created_by_user == null`.
- [ ] Обработчик ответа 403 различает текст про `creator` и существующий 403 про org-доступ; показывает разные toast'ы.
- [ ] Кнопки edit/delete либо дизейблены через `canCurrentUserMutate` (с tooltip), либо кликабельны с обработкой 403.
- [ ] Для sc/rais маркеры скрыты, кнопки всегда активны.
- [ ] Ручное тестирование по 3 (или 4) сценариям из §7.
